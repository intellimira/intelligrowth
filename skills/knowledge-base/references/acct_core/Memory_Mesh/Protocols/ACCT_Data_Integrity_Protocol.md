# ACCT Data-Driven Integrity Protocol (DDIP)

## Design for ACCT's Global Evidence & Fact-Chain Enforcement System

### Purpose:
The Data-Driven Integrity Protocol (DDIP) is a **system-wide enforcement layer** that ensures all outputs, claims, artifacts, and actions produced by ACCT are traceable to verifiable, ground-truth data sources. It prevents hallucinated identifiers, fabricated evidence, and unverified claims from propagating through the operational pipeline.

This protocol was born from a live incident where LLM-generated URLs (hallucinated Reddit post IDs) were passed downstream as if they were real, breaking the entire engagement pipeline. The DDIP codifies the lesson: **"If you can't trace it, you can't claim it."**

### Core Axiom:
> **No output field that represents an external identifier, URL, metric, or factual claim may be generated by an LLM. It must be passed through from a verified source-of-record.**

---

### Core Components:

1.  **The Source-of-Record Registry (SoRR):**
    *   **Objective:** Define which data fields are "Sacred" (must come from verified sources) vs "Synthetic" (can be LLM-generated).
    *   **Sacred Fields (NEVER LLM-Generated):**
        *   URLs, post IDs, user handles, email addresses
        *   Financial figures, transaction IDs, payment links
        *   Timestamps of external events
        *   API keys, tokens, credentials
        *   File paths referencing external systems
    *   **Synthetic Fields (LLM-Generated Allowed):**
        *   Hook text, narrative summaries, strategy briefs
        *   Blueprint descriptions, plan-of-attack prose
        *   Tags, categories, classifications
        *   Internal scoring (PIS, friction scores)

2.  **The Integrity Gate (Runtime Validator):**
    *   **Objective:** A Python module (`data_integrity_gate.py`) that validates all outbound data before it leaves the pipeline.
    *   **Validation Rules:**
        *   **URL Structure Check:** Verify URLs match expected platform patterns (e.g., `reddit.com/r/[sub]/comments/[id]/`).
        *   **Sacred Field Lineage:** Confirm that every Sacred field in an output CSV can be traced back to a row in the Source-of-Record (e.g., `PLS_PROBLEM_MANIFEST.csv`).
        *   **Duplicate Detection:** Flag if the same identifier appears in multiple incompatible contexts.
        *   **Null/Placeholder Detection:** Block fields containing `mock`, `test`, `placeholder`, `YOUR_`, `example.com`.
    *   **Enforcement Mode:**
        *   `STRICT`: Block pipeline execution and alert HITL if any violation is detected.
        *   `AUDIT`: Log violations but allow execution to continue (for development/testing).

3.  **The Evidence Chain:**
    *   **Objective:** Every claim made in an outreach log, report, or HITL alert must link back to a verifiable artifact.
    *   **Chain Structure:**
        *   `Crawler Output` → `PLS_PROBLEM_MANIFEST.csv` (Ground Truth)
        *   `Manifest Row` → `Engagement Hook` (Synthetic text, but Sacred URL passthrough)
        *   `Engagement Log` → `HITL Email` (All Sacred fields inherited, never regenerated)
    *   **Audit Trail:** Every row in `B_LINE_OUTREACH_LOG_REDDIT.csv` must contain a `source_manifest_row_id` field linking it back to the original manifest entry.

4.  **The Watchdog Integration:**
    *   **Objective:** Wire DDIP violations into the existing [[ACCT_Telemetry_Watchdog]] system.
    *   **Alert Triggers:**
        *   Any Sacred field mutation detected between pipeline stages.
        *   Any URL that fails HTTP HEAD validation (returns 404/301).
        *   Any outbound email containing unverified claims.
    *   **Intervention Actions:**
        *   **Pipeline Halt:** Stop the current orchestrator cycle.
        *   **Commander Notification:** Dispatch a `[INTEGRITY_BREACH]` alert via Neural Relay.
        *   **Quarantine:** Move the offending data row to a `QUARANTINE` status in the manifest.

### Contextual Value:
The DDIP is the "Immune System" of ACCT. Without it, the system's credibility collapses the moment a single hallucinated URL reaches a prospect. Every claim must be backed by data, every link must resolve, and every metric must trace to its origin. This is MIRA's core logic: **Data-Driven, Evidence-First, Always.**
