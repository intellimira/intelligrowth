🧠 THE PAINFUL FRAMEWORK
P - PRESSURE
Problem intensity. Measures stress or disruption. "Does it actually hurt?"
A - ADVANTAGE
Upside of solving. Saved time, revenue, or new capabilities. "What is gained?"
I - IMMEDIACY
Urgency. How bad do they need it NOW? Market priority level.
N - NECESSITY
Unavoidability. Must fix to avoid loss or failure. "Can they ignore it?"
F - FREQUENCY
Daily (10) to Rare (0). Frequent problems compound pain exponentially.
U - UNRESOLVED
Fallout if ignored. Cost of doing nothing: lost revenue or risk.
L - LAPSE
Competition gap. Are existing tools FULLY solving this? Room for unbundling.
🏰 MOAT: STRUCTURAL DEFENSE
M - Market Angle: Unpopular segment, Risk removal, or Cognitive load unbundling.
O - Operator Advantage: Skin in the game. Using lived edge cases to build better defaults.
A - Adoption Ease: Zero-setup, "Stupidly Simple" first wins within 60 seconds.
T - Traction Gravity: Sticky logic where leaving means losing history, data, or automation.
💡 TACTICAL HINTS
Rule of 10X: Never build a solution unless the PAINFUL score indicates the benefit is 10X the effort to switch.
The Mom Test: If you can't describe the pain without using "features," you don't understand the problem yet.
Zero-Capital: If it costs more than £0 to build the MVP, re-architect using Tier 0 local-first logic.
